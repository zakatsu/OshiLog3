# OshiLog ver2 要件定義書

## 1. サービス概要

### 1.1 コンセプト

OshiLog ver2は、推し活の「現地参戦ログ」を1つのURLで見せられる1ページ型Webアプリである。

Xのbio欄に参戦履歴を文字で詰め込む代わりに、日付・イベント名・会場・都市をタイムラインとマップで整理して表示する。

ver2では、Firebaseの2種類のデータベースを同じアプリ内で使い分けることを学習目的に含める。参戦ログ本体はFirestoreに保存し、ログごとのコメントはRealtime Databaseに保存する。

### 1.2 ver2の位置づけ

ver1の想定は、静的HTMLとブラウザ内保存を使ったローカルプロトタイプだった。ver2ではFirebaseを導入し、別端末・別ブラウザからも同じ参戦ログを再表示できる構成にする。

---

## 2. 解決したい課題

Xのbio欄には、以下のような参戦ログが書かれているケースがある。

```text
5/10 xxペンミ｜5/16 福岡｜6/1 横浜｜7/20 リリイベ
```

しかし、bio欄に書く方法には課題がある。

- 日付・会場・イベント名が詰まりすぎて見づらい
- 書き方が人によってバラバラ
- 過去の参戦履歴を振り返る一覧性が弱い
- 遠征や会場の広がりが視覚的に伝わらない
- プロフィールとしては情報量が多すぎる

そこで、Xのbio欄にはURLだけを置き、詳細な参戦ログは専用ページで見せる。

## 3. 目指す体験

### 3.1 一言でいう体験

**その人の推し活の現地履歴が、年表と地図で静かに伝わるページ。**

### 3.2 重要な体験価値

1. **軽い**  
   アプリをインストールせず、URLを開くだけで見られる。

2. **見やすい**  
   bio欄の文字列ではなく、日付・場所・イベント名が整って表示される。

3. **振り返りやすい**  
   自分がどこに行って、どんな現場を重ねてきたかが見える。

4. **反応を残せる**  
   タイムラインごとに短いコメントを残せる。

5. **学習テーマが明確**  
   FirestoreとRealtime Databaseの違いを、ログ保存とコメント保存の実装を通じて確認できる。

## 4. データベース方針

### 4.1 Firebaseを使う目的

OshiLog ver2では、Firebaseの2種類のデータベースを実際に使い分ける。

- Firestore
- Realtime Database

同じFirebase内のデータベースでも、データの持ち方・得意な用途・APIの考え方が異なるため、1つのアプリ内で役割を分けて理解する。

### 4.2 Firestoreの役割

Firestoreには、参戦ログ本体を保存する。

保存先:

```text
attendanceLogs/{logId}
```

Firestoreを使う理由:

- ドキュメント単位で構造化されたデータを扱いやすい
- 参戦ログの登録・編集・複製・削除と相性がよい
- `userId` や `eventDate` などのフィールドを持たせやすい
- 将来的に条件検索や並び替えを拡張しやすい

Firestoreで扱うデータ例:

```js
{
  userId: "user_001",
  eventDate: "2026-06-08",
  genre: "合同コン",
  eventName: "2024 Show...",
  venueName: "ベルーナドーム",
  city: "東京",
  latitude: 35.6762,
  longitude: 139.6503,
  createdAt: "...",
  updatedAt: "..."
}
```

### 4.3 Realtime Databaseの役割

Realtime Databaseには、参戦ログごとのコメントを保存する。

保存先:

```text
comments/{logId}/{commentId}
```

Realtime Databaseを使う理由:

- JSONツリーとして軽くデータを追加しやすい
- コメントのように追加された内容をすぐ画面へ反映したいデータと相性がよい
- `onChildAdded` で新規コメントを受け取りやすい
- Firestoreのログ本体とは別の流動的なデータとして扱える

Realtime Databaseで扱うデータ例:

```json
{
  "name": "bbb",
  "text": "このステージが激アツだった",
  "createdAt": "2026-06-08T21:02:00+09:00"
}
```

### 4.4 FirestoreとRealtime Databaseの違い

| 観点 | Firestore | Realtime Database |
|---|---|---|
| データ構造 | コレクションとドキュメント | 1つの大きなJSONツリー |
| 今回の用途 | 参戦ログ本体 | ログごとのコメント |
| 向いているデータ | 構造化されたレコード | 即時反映したい軽いデータ |
| 操作のイメージ | ドキュメントを作成・更新・削除 | パス配下にデータを追加 |
| 実装上の特徴 | `getDocs` / `setDoc` / `deleteDoc` | `push` / `set` / `onChildAdded` |

## 5. ページ構成

### 5.1 1ページ構成

ユーザーが使う画面は、直下の `index.html` 1つだけ。

ページ表示時にFirebaseを初期化し、Firestoreから参戦ログを読み込み、Realtime Databaseからコメントを購読する。

### 5.2 ページ内セクション

1. プロフィールヘッダー
2. 登録・編集フォーム
3. マップ
4. 参戦ログ年表
5. タイムラインごとのコメントパネル

## 6. 画面要件

## 6.1 プロフィールヘッダー

### 表示項目

- 表示名、またはページタイトル
- 「参戦ログのページ」であることが分かる短いラベル

### UI方針

- Linktreeのように縦にすっきり見せる
- 個人情報を過度に載せない
- 情報量を増やしすぎない
- 「参戦ログのページ」であることが直感的に分かるコピーを置く

### 表示例

```text
Oshi attendance log
現地参戦ログ
```

---

## 6.2 登録・編集フォーム

### 目的

参戦ログを同じページ内で登録・編集できるようにする。

### 入力項目

- 日付
- 種別
- イベント名
- 会場名
- 都市

### UI方針

- PC幅では1行で入力できる
- モバイル幅では自然に折り返す
- 緯度経度はユーザーに入力させない
- 編集時は見出しを `Edit` に変更し、選択中レコードの値をフォームへ読み込む
- 編集時は `更新する` と `キャンセル` を表示する

### 種別

- 単独ライブ
- 合同コン
- ファンミ
- イベント
- 収録参加
- ヨントン

---

## 6.3 マップ

### 目的

その人がどの場所に参戦してきたかを、直感的に見せる。

### 表示要件

- 登録された会場または都市をピンで表示する
- 同じ場所に複数ログがある場合は、1つのピンにまとめる
- ピンをタップすると、該当するログ名を簡単に表示する
- 年表のログをタップすると、対応するピンがハイライトされる
- ズームアウトボタンで全体表示に戻せる
- ズームインボタンで選択中ログ、未選択時はタイムライン先頭ログへズームできる

### UI方針

- 地図を主役にしすぎない
- ピンの色やアニメーションに少しこだわる
- スマホで見ても重くならないことを優先する

### 都市座標

正確な会場住所ではなく、都市名に紐づく緯度経度を `cities.json` から自動判定して保存する。住所検索APIや外部ジオコーディングは後回しにする。

読み込みに失敗した場合に備えて、`app.js` には最小限のフォールバック座標を残す。

---

## 6.4 参戦ログ年表

### 目的

参戦履歴をbio欄の文字列ではなく、読みやすい時系列として見せる。

### 表示項目

- 日付
- 種別
- イベント名
- 会場名
- 都市
- コメント数

### UI方針

- 年表形式で縦に並べる
- 年ごとに区切る
- 1ログはタイムライン行として表示する
- タップすると該当ログが選択状態になり、フォームが編集状態になる
- 選択変更時は、マップが一度引いてから対象ピンへズームする
- キーボード操作で前後のログへ移動できる
- 複製できる
- リセットできる
- 詳細ページには遷移しない
- 1レコード内のイベント名・会場名は改行せず、長い場合は省略表示する

### キーボード操作

- `↑` / `←`: 前のログを選択
- `↓` / `→`: 次のログを選択
- `Enter` / `Space`: 現在のログを編集対象にする

---

## 6.5 コメントパネル

### 目的

参戦ログごとに、短い反応や共有コメントを残せるようにする。

### 表示項目

- コメント投稿者名
- コメント本文
- 投稿日時

### 入力項目

- 名前
- コメント

### UI方針

- タイムライン行のコメントボタンから開閉する
- コメントは1行入力にし、タイムライン内で表示バランスを崩さない
- PC幅では名前・コメント・送信ボタンを横並びにする
- コメント一覧とフォームはタイムラインカードの幅内に収める
- まだコメントがない場合は「まだコメントはありません。」と表示する

## 9. 非機能要件

### 9.1 デバイス

- スマートフォン最優先
- PCでも崩れない
- 1カラム中心のレスポンシブデザイン
- PC幅ではマップとタイムラインを横並びにできる

### 9.2 表示速度

- 1ページの初期表示を軽くする
- 画像や重いアニメーションは使いすぎない
- ログ数100件程度までは破綻しない
- コメントはログごとに必要な範囲で表示する

### 9.3 プライバシー

- 過去の行動履歴が公開される可能性があるため、公開前に注意喚起する
- 個人名、SNS ID、行動の詳細を載せすぎない
- コメント欄は公開用途では荒らし対策と認証を検討する
- 将来的には、会場名非表示・都市のみ表示などを検討する

### 9.4 保守性

- ログデータはイベント単位で保持する
- コメントはログ本体に埋め込まず、Realtime Databaseに分離する
- 都市座標マスタは `cities.json` で管理する
- Firebase設定は `firebase-config.js` に分離し、GitHubにはコミットしない

## 11. UI方針

### 11.1 目指す印象

- シンプル
- 余白がある
- 事実が整って見える
- 参戦履歴がきれいに見える
- 過剰な自己表現やSNS感を出さない

### 11.2 UIのこだわりポイント

#### A. 年表の美しさ

単なるリストではなく、日付がリズムよく並ぶ年表にする。日付・イベント名・会場名・都市の情報階層を明確にする。

#### B. 地図との連動

ログをタップした瞬間にピンが反応することで、「現地の軌跡」が視覚的に分かる。

#### C. コメント欄の収まり

コメント一覧と入力フォームは、タイムラインカードの中に収める。コメント入力は1行にし、名前欄・コメント欄・送信ボタンの高さを揃える。

#### D. 個人情報を載せすぎない

公開ページに個人名、SNS ID、行動の詳細、感情的なコメントを載せすぎない。基本は「いつ・どこで・何に参加したか」という事実のログに絞る。

## 13. 初期実装の割り切り

### 13.1 登録方法

管理画面は作らず、ページ上部に簡易フォームを置く。

登録・編集・複製・リセットはすべて同一ページ内で行う。

### 13.2 データ保存

ver2では `localStorage` ではなくFirebaseに保存する。

- ログデータ用JSONファイルは使用しない
- 都市座標マスタのみ `cities.json` を使用する
- 参戦ログはFirestoreに保存する
- コメントはRealtime Databaseに保存する
- Firebase設定がない場合は保存できない

## 16. ver2の一言定義

Firestoreに参戦ログを保存し、Realtime Databaseにコメントを保存することで、Firebaseの2種類のデータベースの違いを学びながら、推し活の現地履歴を1枚の年表＋マップで振り返れるWebアプリ。
